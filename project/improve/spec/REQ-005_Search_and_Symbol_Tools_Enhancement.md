# REQ-005: 検索・シンボルツール強化（improve feature）要件定義

**要件ID**: REQ-005
**Feature**: improve
**作成日**: 2026-05-01
**対象**: search_code / find_symbol_definition / find_symbol_references / find_references
**ステータス**: 起草中
**関連文書**: REQ-001, REQ-002, REQ-003

---

## 1. 概要

Swift-Selena の検索系・シンボル系ツールを強化し、grep 用途として実装現場の要求水準を満たすと同時に、LSP ベースの意味検索という Selena 固有の優位性をさらに伸ばす。

本 Feature の対象は既存ツールの**強化**であり、新規ツール追加は最小限とする。

---

## 2. 要求背景

### 2.1 現状の課題

#### 課題1: search_code の結果量が制御できない

- `search_code` は全マッチを返す
- 巨大プロジェクトで一般語（`func`, `let`, `View` 等）を検索すると返却サイズが膨大化
- 呼び出し側（MCP クライアント）の上限を超えた場合、結果が外部ファイル化されて**実質 grep として使えない**
- 「どのファイルに存在するか」だけ知りたいケースが多数（全マッチ列挙は不要）

#### 課題2: 結果が機械処理しにくい

- 出力はプレーンテキストのみ
- 呼び出し側（AI/Agent）が結果を加工・絞り込みする際、テキストパースが必要
- ファイルパス・行番号・マッチ内容を構造化された形で取得できない

#### 課題3: ファイルパターンの表現力不足

- `file_pattern` は単一 glob のみ
- テスト・モック・自動生成コード等を**除外したい**頻出シナリオに対応できない
- 複数の拡張子・ディレクトリを同時に指定できない

#### 課題4: シンボル定義検索の曖昧性

- `find_symbol_definition` は完全一致するシンボルを全件返す
- 同名シンボルが**異なるスコープ**（ネスト型・extension・別モジュール）に複数存在する場合、どれが目的のものか判別できない
- 種別（struct/class/protocol 等）による絞り込みができない
- 親スコープ・extension 対象の情報が出力に含まれず、呼び出し側で区別できない

#### 課題5: シンボル名による正確な参照検索ができない

- `find_symbol_references`（LSP版・REQ-002）は **ファイルパス + 行番号 + 列番号** の指定が必須
- 「シンボル名だけ」では呼び出せない
- `find_references`（テキストベース・REQ-003）は単語境界で絞り込むが、同名異シンボルを区別できない
- 「`Theme` の使用箇所を正確に」というユースケースが既存ツールでは満たせない

### 2.2 解決方針

| 課題 | 解決方針 |
| ---- | -------- |
| 結果量制御 | 出力モード（一覧種別）と件数上限を導入 |
| 機械処理性 | 既存テキスト出力は維持し、構造化データを併記 |
| ファイルパターン | include/exclude を配列で受け取れるようにする |
| シンボル曖昧性 | 種別フィルタと所属スコープ情報を追加 |
| 名前ベース参照検索 | 既存の参照検索ツールに「シンボル名解決」入口を追加 |

## 3. 用語定義（グロッサリー）

| 用語 | 意味 |
| ---- | ---- |
| 出力モード | 検索結果の表現形式（マッチ詳細/ファイル一覧/件数のみ等） |
| マッチ | 検索条件に合致した行・シンボル・ファイル等の単位 |
| 件数上限 | 結果として返す最大件数。超過分は省略され省略があった旨を伝える |
| 含めるパターン | 検索対象に含めるファイルを特定する glob |
| 除くパターン | 検索対象から除外するファイルを特定する glob |
| シンボル種別 | struct / class / enum / protocol / actor / function / variable / typealias / extension の 9 区分。利用者が指定する値は小文字スネーク表記（既存ツールが返す表示用 kind 値（`Class` / `Struct` / `Actor` 等）との対応関係は TBD-005 で確定する） |
| 所属スコープ情報 | シンボルの所属を示す総称。**親スコープ名**（ネストされている親型の名前）／**extension 対象型**（extension 内で定義された場合の対象型）／**モジュール名**（検出可能な場合）の 3 要素から構成される。同名シンボルの区別に用いる |
| シンボルインデックス | SourceKit-LSP が型情報を考慮した参照解決のために構築・参照する索引。本要件では具体的な実装形式（永続化方式・格納先等）には踏み込まず、参照解決の前提として「構築済みかどうか」のみを扱う |
| LSP 精度 | シンボルインデックスに基づいて型情報を考慮した参照集合（SourceKit-LSP の `textDocument/references` と等価な集合）を返す精度 |
| テキスト精度 | テキスト一致のみで参照を抽出する精度（同名異シンボルを区別できない） |
| 構造化結果 | フィールド名で識別可能な機械処理向けデータ表現 |

## 4. 要件一覧

### 4.1 検索結果の量制御（search_code）

**要件**:
`search_code` は、結果サイズが膨大化する場合に備え、利用者が**結果の表現粒度**と**件数上限**を選択できる。

**選択可能な出力モード**:

| モード | 用途 | 結果の内容 |
| ------ | ---- | ---------- |
| マッチ詳細 | 既定。従来どおりの動作（後方互換） | ファイル・行番号・該当行内容 |
| ファイル一覧 | 「どこにあるか」だけ知りたい | マッチを含むファイルの一覧（重複排除） |
| 件数のみ | 量だけ把握したい | マッチ数とファイル数 |

**件数上限の適用単位**:

出力モードごとに上限の単位が異なる。利用者は単位を意識せずモードに応じた最大件数を指定する。

| 出力モード | 件数上限の単位 | 上限超過時に省略される対象 |
| ---------- | -------------- | -------------------------- |
| マッチ詳細 | マッチ行 | 上限件数を超えたマッチ行 |
| ファイル一覧 | ファイル（重複排除後） | 上限件数を超えたファイル |
| 件数のみ | （上限指定の対象外。指定されても集計値には影響しない） | 該当なし |

**件数上限の境界値**:
- 件数上限の最小値: 1（0・負数・整数以外は入力エラー）
- 件数上限の既定値: 未指定時は全件返す。ただし結果量制御の動機（§2.1 課題1）と矛盾しうるため、TBD-006 として既定値の見直しを留保する

**省略の明示方法**:
- 上限を超えた場合、上限件数まで返し、**省略があった旨**を結果に明示する
- テキスト出力には省略があった旨のメッセージを含める
- 構造化結果には省略フラグ（§4.2）を `true` で含める

**受入基準**:
- 出力モードを選択できる
- 既定値は「マッチ詳細」（後方互換）
- 既定値「マッチ詳細」のテキスト出力は、既存ツール（マッチ件数ヘッダ + `<file>:<line>: <content>`（前後空白除去））と同等の整形を維持する（§4.6 後方互換）
- 件数上限を 1 以上の整数で指定できる
- 上限超過時、テキスト出力と構造化結果の双方で省略されたことが明示される
- §4.2 構造化結果のフィールド値（ファイルパス・行番号・マッチ行内容）は「マッチ詳細」テキスト出力と意味が一致する

**未確定事項への参照**: TBD-001（出力モード既定値の見直し）／TBD-006（件数上限既定値の見直し）

### 4.2 構造化された検索結果（search_code 等）

**要件**:
検索結果は、人間が読める形式（既存）に加えて、**フィールド名で識別可能な構造化データ**としても取得できる。

§4.2 は全対象ツール共通の最低限フィールドのみを定義する。各ツール固有の追加フィールドは §4.4 / §4.5 で個別に定義する。

**共通フィールド（全対象ツール最低限）**:

| フィールド | 内容 |
| ---------- | ---- |
| ファイルパス | マッチが見つかったファイルの絶対パス |
| 行番号 | マッチした行の番号（1-indexed） |
| マッチ行内容 | 行のテキスト内容（前後空白除去） |
| 検索結果総数 | 上限適用前の総数 |
| 省略フラグ | 件数上限による省略の有無 |

**ツール固有の追加フィールド**:

| 対象ツール | 追加フィールド | 定義箇所 |
| ---------- | -------------- | -------- |
| find_symbol_definition | 種別・所属スコープ情報（親スコープ名 / extension 対象型 / モジュール名） | §4.4 |
| シンボル名による参照検索（§4.5） | 所属スコープ情報・LSP 利用フラグ・フォールバック発生時の追加情報 | §4.5 |
| find_references / find_symbol_references（既存） | 所属スコープ情報（呼び出し結果に区別が必要な場合） | §4.5 |

**対象ツール**:
- search_code
- find_symbol_definition（4.4 と連動）
- find_references / find_symbol_references（4.5 と連動）

**受入基準**:
- 既存のテキスト出力は維持される（後方互換）
- 構造化結果が同時に取得可能である
- 利用者がパースせずにフィールドへアクセスできる
- 共通フィールドは全対象ツールで取得できる
- ツール固有の追加フィールドは §4.4 / §4.5 で定義された内容を満たす

**注**: 提供方法（MCP の structuredContent 併記等）は設計フェーズで決定する（How）。

**未確定事項への参照**: TBD-004（構造化結果の表現形式）

### 4.3 ファイルパターンの複数指定（search_code）

**要件**:
`search_code` のファイル絞り込みは、**含めるパターン**と**除くパターン**をそれぞれ複数指定できる。

**典型的な利用シナリオ**:
- テスト・モック・自動生成コードを除外して本実装のみを検索
- 複数の拡張子・ディレクトリを同時対象にする

**指定方法**:
- 含めるパターン: 0 個以上の glob（空配列は未指定と等価）
- 除くパターン: 0 個以上の glob（空配列は未指定と等価）
- 既定: 含めるパターン未指定時は Swift ファイル全体（既存挙動）

**評価規則**:
- 含めるパターンに合致し、かつ除くパターンに合致しないファイルのみが対象となる
- 複数の含めるパターンは OR 結合
- 複数の除くパターンは OR 結合
- 含めるパターン・除くパターンの両方に合致するファイルは**除くパターンを優先**して除外する
- 不正な glob 構文を検出した場合は入力エラーとして利用者に明示する

**既存パラメータとの併存**:
- 既存の単一指定パラメータ（従来の `file_pattern` 相当）は維持し、後方互換として引き続き動作する
- 既存の単一指定と新規の含めるパターン配列が同時に指定された場合の優先順位は TBD-007 として §7 に記録し、設計フェーズで確定する
- 配列要素数の上限値（極端な指定でメモリ等を圧迫しないための上限）は要件レベルでは定めず、設計フェーズで上限方針を決める（要件側では下限 0 と既定挙動のみを規定する）

**受入基準**:
- 含めるパターン・除くパターンを配列で受け取れる
- 既存の単一指定（後方互換用）も引き続き動作する
- 既定の挙動は従来と一致する
- 不正な glob 入力時はエラーが利用者に明示される
- 含めるパターンと除くパターンが衝突した場合は除くパターンを優先する

**スコープ外**:
- 論理層（Domain/UI/Infrastructure 等）による絞り込み機能は本 Feature に含めない
- 理由: Selena 自体がプロジェクト構造を理解する基盤を持たないため、別 Feature として整理する

### 4.4 シンボル定義検索の絞り込み強化（find_symbol_definition）

**要件**:
`find_symbol_definition` は、同名シンボルが複数存在する場合に**種別**と**所属スコープ情報**で結果を絞り込み・識別できる。

#### 4.4.1 種別フィルタ

**指定可能な種別**（小文字スネーク表記。§3 用語定義に準拠）:
- struct / class / enum / protocol / actor
- function / variable / typealias / extension

**動作**:
- 種別を指定した場合、その種別のシンボルのみを返す
- 種別未指定時は全種別を返す（既存挙動を継承し、優先順序は REQ-003 §2.3 v0.6.5+ で定められた挙動を維持する）
- 未定義の種別文字列が指定された場合は入力エラーとして利用者に明示する

#### 4.4.2 所属スコープ情報の付与

**結果に含める情報（§3 用語定義「所属スコープ情報」の構成要素）**:

| 構成要素 | 内容 |
| -------- | ---- |
| 親スコープ名 | ネストされている親型の名前（存在する場合） |
| extension 対象型 | extension 内で定義された場合の対象型 |
| モジュール名 | 検出可能な場合 |

**区別すべきケース（受入基準で参照）**:
- (i) ルート定義の `Button`
- (ii) ネスト型 `enum Foo { struct Button }` の `Foo.Button`
- (iii) `extension Foo { struct Button }` の `Foo.Button`（extension 由来）

**受入基準**:
- 種別フィルタを指定できる
- 種別未指定時は REQ-003 §2.3 v0.6.5+ の優先順序を維持しつつ全種別を返す
- 未定義の種別が指定された場合はエラーが利用者に明示される
- 結果に**所属スコープ情報**（親スコープ名・extension 対象型・モジュール名）が含まれる
- 所属スコープ情報により、上記 (i)(ii)(iii) の 3 ケースを互いに区別可能である

**未確定事項への参照**: TBD-002（モジュール名取得の精度）／TBD-005（小文字スネーク指定値と既存表示用 kind 値の対応）

### 4.5 シンボル名による正確な参照検索

**要件**:
シンボルの**名前だけ**を入力として、LSP 精度（§3 用語定義参照）の参照検索が実行できる。

**前提条件（既存ツールの実装状況）**:
- 本節は既存ツール `find_symbol_references`（LSP 精度・位置指定）／`find_references`（テキスト精度・名前指定）が利用可能であることを前提とする
- 本 Feature 開始時点でこれら 2 ツールの実装状況が要件と一致しない場合は、本体実装の取り込み判断を TBD-008 として §7 に記録し、設計フェーズで A 案（本 Feature にスコープを拡げて本体も実装する）または B 案（本 Feature では §4.5 をスコープ外に送り、後続 feature で実装する）のいずれかを確定する
- A/B いずれを採るかは設計判断であり、本要件では「§4.5 を成立させるためにどちらかの確定が必要」という要請のみを記述する

**現状ツールの位置付け**:

| ツール | 入力 | 精度 | 制約 |
| ------ | ---- | ---- | ---- |
| find_symbol_references（LSP） | ファイルパス + 行番号 + 列番号 | LSP 精度 | 位置指定が必須 |
| find_references（テキスト） | シンボル名 | テキスト精度（同名異シンボルを区別できない） | LSP 不要 |
| find_type_usages | 型名 | テキスト精度（型のみ） | 型限定 |

**本要件で満たすこと**:
- 「シンボル名」を入力として、まずシンボル定義を解決し、その位置情報をもとに LSP 精度の参照検索を行うフローを成立させる
- 同名異シンボルが複数存在する場合、利用者が**どの定義の参照を求めるか**を後続呼び出しで指定できる、または単一呼び出しの結果が**所属スコープ情報単位で分類**されている

**選択手段（複数候補時のインタラクション）**:
- 単一呼び出しで結果を返す場合: 所属スコープ情報ごとにグルーピングされた結果を返し、利用者は結果を見てどの候補を採用するか判断する
- 後続呼び出しで絞り込む場合: 候補一覧（各候補に位置情報または所属スコープ識別子を付与）を返し、利用者は採用したい候補の識別情報を再度パラメータとして渡して再呼び出しする
- 上記のいずれの方式を採用するかは設計フェーズで確定する（TBD-003 と関連付ける）

**受入基準**:
- シンボル名のみで参照検索が呼び出せる
- LSP 利用可能時は LSP 精度の結果（§3 用語定義に準拠）を返す
- LSP 利用不可時、本要件で導入する経路は内部で `find_references` 相当のテキストベース検索ロジックを呼び出してフォールバックする（既存 `find_references` の出力契約は変更しない）
- フォールバック発生時、利用者に対して結果へ次の情報をすべて含めて明示する:
  - (a) フォールバック発生フラグ
  - (b) 原因区分（例: ビルド不可 / シンボルインデックス未構築 / LSP セッション切断 等。最低限「LSP 利用可否」と「原因のユーザー観察可能な区分」を含める）
  - (c) 代替案（テキスト精度結果のみで判断するか、ビルド完了およびシンボルインデックス構築後に再呼び出しするか等の案内）
- 同名異シンボルが複数ある場合、採用した方針（単一呼び出し方式 / 後続呼び出し方式）に応じて以下のいずれかが成立する:
  - 単一呼び出し方式を採用した場合: 結果が所属スコープ情報ごとに識別可能である
  - 後続呼び出し方式を採用した場合: 最初の呼び出し結果が候補一覧として所属スコープ情報単位で識別可能である
- LSP セッションが処理途中で切断された場合は、テキスト精度のフォールバックに切り替え、上記 (a)(b)(c) を結果に明示する
- シンボル名解決で 0 件となった場合は空の結果を返す（エラーではない）

**前提条件**:
- LSP 精度の結果は、プロジェクトがビルド可能で、シンボルインデックスが構築されている状態でのみ得られる
- 上記が満たされない場合のフォールバックは精度が劣る旨を結果に明示する（具体的な状態判定方法・シンボルインデックス管理方式は設計フェーズで決定する）

**スコープ外**:
- 既存 `find_symbol_references`（位置指定版）の API 入力仕様は変更しない（後方互換）
- シンボル名入力経路は**別経路**として提供する。具体的な API 表面（既存ツールへの入力モード追加 / 新規ツール追加）は TBD-003 で確定する
- 既存 `find_references`（テキスト版）の出力契約（REQ-003 §2.3）は変更しない

**未確定事項への参照**: TBD-003（API 表面の振り分け）／TBD-008（§4.5 既存前提ツールの本体実装スコープ）

### 4.6 後方互換性

**要件**:
既存の呼び出し方法・出力形式は、本 Feature 実装後も引き続き動作する。

**保証する後方互換**:
- 既存ツールの呼び出し時に新規パラメータを未指定でも、従来と同等の挙動を返す
- 既存の人間可読テキスト出力は維持される
- 既存ツール名・既存パラメータ名は変更しない

**「追加情報の混在」の境界（検証可能な範囲を構造的に限定）**:
- 既存出力の各行頭フォーマット（例: `<file>:<line>: <content>`）は変更しない
- 追加情報は**新規行・新規ブロック・末尾セクション**として付加する形でのみ許容する
- 既存行内に新たな区切り文字や追加トークンを混在させない

**受入基準**:
- 既存の呼び出しコードを変更せずに動作する
- 既存出力の行頭フォーマット（例: `<file>:<line>: <content>`）が変更されないことを確認できる
- 追加情報が新規行・新規ブロック・末尾セクションとしてのみ付加される
- Swift-Selena 自身の既存テストが破壊的変更なく通過する（既存テキスト出力に依存するテストを含む）

---

### 4.7 非機能要件

#### 4.7.1 性能

**要件**:

| 項目 | 目標 |
| ---- | ---- |
| 件数上限指定時の応答 | 件数上限を指定した場合、上限を超える結果が存在しても利用者が体感できる遅延なく結果を返す |
| 構造化結果生成のオーバーヘッド | 既存テキスト出力のみを取得した場合と比較して、利用者が体感できる遅延を生じない |

**注**:
- 具体的な数値目標は本 Feature 完了時に運用計測で検証する。設計段階で「達成保証」は行わない（spec_priorities_spec §2 に従う）
- 性能目標を達成するための実装方針は設計書（DES-103 等）で扱う
- 検証手段（運用計測の観測項目・観測方法）は設計フェーズで決定し、計画書に検証タスクとして記録する

#### 4.7.2 可用性

**要件**:
- LSP 利用不可時にもグレースフルデグレード（テキスト精度のフォールバック）が成立する（REQ-002 NFR-LSP-001 と整合）
- フォールバック発生時、利用者は §4.5 受入基準で定めた情報により発生事実を識別できる

#### 4.7.3 運用性

**要件**:
- 利用者が結果から状況（LSP 利用可否・フォールバック発生・件数上限による省略の有無）を把握できる
- 不正入力（正規表現・glob・件数上限・種別フィルタ）に対して原因と修正案を含むエラーメッセージを返す（既存 v0.6.5+ の方針と整合）

---

### 4.8 異常系要件

**要件**:
正常系と並行して、以下の異常系・境界条件における挙動を本 Feature の要件として確定する。

| 区分 | 状況 | 要件 |
| ---- | ---- | ---- |
| 入力検証 | 正規表現の構文エラー（§4.1 search_code の pattern） | 入力エラーとして原因と修正案を利用者に明示する |
| 入力検証 | glob の構文エラー（§4.3 含めるパターン・除くパターン） | 入力エラーとして原因と修正案を利用者に明示する |
| 入力検証 | 件数上限が 0・負数・整数以外（§4.1） | 入力エラーとして利用者に明示する（最小値は 1） |
| 入力検証 | 未定義のシンボル種別文字列（§4.4.1） | 入力エラーとして利用者に明示する |
| 入力検証 | 含めるパターンと除くパターンが両方未指定 | 既定挙動（含めるパターン未指定 = Swift ファイル全体）を適用する |
| 入力検証 | 含めるパターンと除くパターンが同一ファイルに同時マッチ | 除くパターンを優先する（§4.3 評価規則と整合） |
| シンボル解決 | §4.5 でシンボル名解決が 0 件 | 空の結果を返す（エラーではない。REQ-003 §2.3 find_references の挙動と整合） |
| シンボル解決 | §4.5 で同名シンボルが多数件（具体的な閾値は設計フェーズで決定） | §4.5 の選択手段に従って所属スコープ単位で分類した結果を返す。件数上限（§4.1）が指定されている場合は併用する |
| LSP | LSP セッションが処理途中で切断 | テキスト精度のフォールバックに切り替え、§4.5 受入基準 (a)(b)(c) を結果に明示する |
| 永続化 | キャッシュ破損・読み込み失敗 | キャッシュを無視して通常解析を実行し、利用者に状況を明示する（既存ツール挙動と整合） |

**受入基準**:
- 上記表の各状況において、それぞれの「要件」列に記載した挙動が成立する
- 入力検証エラーは利用者がエラー内容を見て修正できるメッセージを含む

## 5. スコープ外（本 Feature では扱わない）

| 項目 | 理由 / 振り分け先 |
| ---- | ----------------- |
| コメント・文字列リテラル除外検索 | LSP ではなく SwiftSyntax の責務。性能トレードオフが大きく、別 Feature として独立評価が必要 |
| 論理層による絞り込み（layer enum 等） | Selena がプロジェクトの層構造を理解する基盤を持たないため、別 Feature とする |
| ファイル本文読み取りツール | 本 Feature の対象外。標準 Read 系の責務として明確に分離する |
| 既存ツールのデフォルト破壊的変更 | 後方互換維持の方針に反する |
| 構造品質の数値目標化 | spec_priorities_spec §3 に従い禁止 |

### 6.2 矛盾しないことの確認

- 本要件は既存ツールの**追加機能**であり、既存要件の挙動を打ち消すものではない
- 後方互換維持（4.6）により、既存要件で定義された挙動はすべて保持される
- REQ-003 §2.3 find_symbol_definition v0.6.5+ の優先返却挙動（Class/Struct/Enum/Protocol/Actor を優先）は本要件によって変更されない（§4.4.1 動作で明示済み）
- REQ-003 §2.3 find_references の出力契約（参照が見つからない場合は空のリストを返す等）は本要件によって変更されない（§4.5 スコープ外で明示済み）

### 6.3 既存ツール実装状況との関係

- 本要件 §4.5 は既存ツール `find_symbol_references` / `find_references` が利用可能であることを前提とする
- 本 Feature 開始時点で当該ツールの実装が要件と一致しない場合、本体実装の取り込み判断は TBD-008 で扱う

## 8. 成功基準

| 観点 | 基準 |
| ---- | ---- |
| grep 用途 | 一般語検索でも結果サイズを利用者側で制御でき、外部ファイル化を回避できる |
| 機械処理 | 呼び出し側がテキストパースなしで結果を扱える |
| 絞り込み | テスト・モックを除外した本実装検索が単一呼び出しで完結する |
| 曖昧性解消 | 同名シンボルが複数あっても、所属スコープ情報により目的の定義を識別できる |
| 名前ベース参照検索 | シンボル名のみの入力で参照箇所が取得できる（LSP 利用時は LSP 精度の結果が返る） |
| 後方互換 | 既存呼び出し側を変更せずに動作する |
| 異常系 | 入力検証エラー・LSP 切断・キャッシュ破損等の異常系で、利用者が状況を把握できる結果が返る |
