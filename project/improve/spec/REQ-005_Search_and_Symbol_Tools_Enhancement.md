# REQ-005: 検索・シンボルツール強化（improve feature）要件定義

**要件ID**: REQ-005
**Feature**: improve
**作成日**: 2026-05-01
**対象**: search_code / find_symbol_definition（`find_symbol_references` / `find_references` は本 Feature ではスコープ外。§4.5 参照）
**ステータス**: 起草中
**関連文書**: REQ-001, REQ-002, REQ-003

---

## 1. 概要

Swift-Selena の検索系・シンボル系ツールを強化し、grep 用途として実装現場の要求水準を満たすと同時に、LSP ベースの意味検索という Selena 固有の優位性をさらに伸ばす。

本 Feature の対象は既存ツールの**強化**であり、新規ツール追加は最小限とする。

---

## 2. 要求背景

### 2.1 現状の課題

#### 課題1: search_code の結果量が制御できない

- `search_code` は全マッチを返す
- 巨大プロジェクトで一般語（`func`, `let`, `View` 等）を検索すると返却サイズが膨大化
- 呼び出し側（MCP クライアント）の上限を超えた場合、結果が外部ファイル化されて**実質 grep として使えない**
- 「どのファイルに存在するか」だけ知りたいケースが多数（全マッチ列挙は不要）

#### 課題2: 結果が機械処理しにくい

- 出力はプレーンテキストのみ
- 呼び出し側（AI/Agent）が結果を加工・絞り込みする際、テキストパースが必要
- ファイルパス・行番号・マッチ内容を構造化された形で取得できない

#### 課題3: ファイルパターンの表現力不足

- `file_pattern` は単一 glob のみ
- テスト・モック・自動生成コード等を**除外したい**頻出シナリオに対応できない
- 複数の拡張子・ディレクトリを同時に指定できない

#### 課題4: シンボル定義検索の曖昧性

- `find_symbol_definition` は完全一致するシンボルを全件返す
- 同名シンボルが**異なるスコープ**（ネスト型・extension・別モジュール）に複数存在する場合、どれが目的のものか判別できない
- 種別（struct/class/protocol 等）による絞り込みができない
- 親スコープ・extension 対象の情報が出力に含まれず、呼び出し側で区別できない

#### 課題5: シンボル名による正確な参照検索ができない [本 Feature ではスコープ外]

本課題（旧 §4.5）は本 Feature ではスコープ外確定。理由・経緯は §4.5 / §6.3 を参照。LSP 系参照検索ツールの再導入可否は別途独立 Feature として検討する。

**課題自体は現状残存する**（本 Feature では解決しないが事実として記録）:
- `find_symbol_references`（LSP 精度・位置指定）はファイルパス + 行番号 + 列番号の指定が必須で、シンボル名のみからの呼び出しはできない
- `find_references`（テキスト精度・名前指定）は同名異シンボルを区別できない
- 両ツールとも 2026-05-01 時点で実装が存在しない（§6.1 / §6.3 / §4.5 で参照される事実）

### 2.2 解決方針

| 課題 | 解決方針 |
| ---- | -------- |
| 結果量制御 | 出力モード（一覧種別）と件数上限を導入 |
| 機械処理性 | 既存テキスト出力は維持し、構造化データを併記 |
| ファイルパターン | include/exclude を配列で受け取れるようにする |
| シンボル曖昧性 | 種別フィルタと所属スコープ情報を追加 |
| 名前ベース参照検索 | 本 Feature ではスコープ外（§4.5 / §5 参照） |

## 3. 用語定義（グロッサリー）

| 用語 | 意味 |
| ---- | ---- |
| 出力モード | 検索結果の表現形式。値は「マッチ詳細」「ファイル一覧」「件数のみ」の 3 種類とする |
| マッチ | 検索条件に合致した行・シンボル・ファイル等の単位 |
| 件数上限 | 結果として返す最大件数。超過分は省略され省略があった旨を伝える |
| 含めるパターン | 検索対象に含めるファイルを特定する glob |
| 除くパターン | 検索対象から除外するファイルを特定する glob |
| シンボル種別 | struct / class / enum / protocol / actor / function / variable / typealias / extension の 9 区分。利用者が指定する値は小文字スネーク表記（既存ツールが返す表示用 kind 値（`Class` / `Struct` / `Actor` 等）との対応関係は TBD-005 で確定する） |
| 所属スコープ情報 | シンボルの所属を示す総称。**親スコープ名**（ネストされている親型の名前）／**extension 対象型**（extension 内で定義された場合の対象型）／**モジュール名**（検出可能な場合）の 3 要素から構成される。同名シンボルの区別に用いる |
| LSP 精度 | LSP（SourceKit-LSP の `textDocument/references` 等）を介して型情報を考慮した参照集合を返す精度。本 Feature では §4.5（スコープ外）および §6.1 / §6.3 / §5 の歴史的経緯記述でのみ用いる |
| テキスト精度 | テキスト一致のみで参照を抽出する精度（同名異シンボルを区別できない）。本 Feature では §4.5（スコープ外）および §6.1 / §6.3 / §5 の歴史的経緯記述でのみ用いる |
| 構造化結果 | フィールド名で識別可能な機械処理向けデータ表現 |
| 省略フラグ | 件数上限による省略の有無を構造化結果で示すフィールド |
| 優先返却 | 結果集合のうち特定の区分（例: Class / Struct / Enum / Protocol / Actor）が非該当区分より前に並ぶことを指す。区分内部の序列は規定しない |

## 4. 要件一覧

### 4.1 検索結果の量制御（search_code）

**要件**:
`search_code` は、結果サイズが膨大化する場合に備え、利用者が**結果の表現粒度**と**件数上限**を選択できる。

**選択可能な出力モード**:

| モード | 用途 | 結果の内容 |
| ------ | ---- | ---------- |
| マッチ詳細 | 既定。従来どおりの動作（後方互換） | ファイル・行番号・該当行内容 |
| ファイル一覧 | 「どこにあるか」だけ知りたい | マッチを含むファイルの一覧（重複排除） |
| 件数のみ | 量だけ把握したい | マッチ数とファイル数 |

**件数上限の適用単位**:

出力モードごとに上限の単位が異なる。利用者は単位を意識せずモードに応じた最大件数を指定する。

| 出力モード | 件数上限の単位 | 上限超過時に省略される対象 |
| ---------- | -------------- | -------------------------- |
| マッチ詳細 | マッチ行 | 上限件数を超えたマッチ行 |
| ファイル一覧 | ファイル（重複排除後） | 上限件数を超えたファイル |
| 件数のみ | （上限指定の対象外。指定されても集計値には影響しない） | 該当なし |

**件数上限の境界値**:
- 件数上限の最小値: 1（0・負数・整数以外は入力エラー）
- 件数上限の最大値（上限の上限）: 要件レベルでは具体値を定めず、設計フェーズで決定する。最大値を超えた入力に対する挙動（入力エラー or 内部上限への切り詰め通知）の選択は TBD-010 で確定する。要件としては「最大値を超えた入力に対して、利用者が状況を識別できる結果（入力エラー or 内部上限への切り詰め通知のいずれか）が返される」ことを満たす
- 件数上限の既定値: 未指定時は全件返す。ただし結果量制御の動機（§2.1 課題1）と矛盾しうるため、TBD-006 として既定値の見直しを留保する

**省略の明示方法**:
- 上限を超えた場合、上限件数まで返し、**省略があった旨**を結果に明示する
- テキスト出力には省略があった旨のメッセージを含める
- 構造化結果には省略フラグ（§4.2）を `true` で含める

**受入基準**:
- 出力モードを選択できる
- 既定値は「マッチ詳細」（後方互換）
- 既定値「マッチ詳細」のテキスト出力は、既存ツール（マッチ件数ヘッダ + `<file>:<line>: <content>`（前後空白除去））と同等の整形を維持する（§4.6 後方互換）
- 件数上限を 1 以上の整数で指定できる
- 件数上限未指定時は全件返す（TBD-006 確定後に変更されうる）
- 利用者が指定した件数上限を結果が超過した場合、テキスト出力と構造化結果の双方で省略されたことが明示される
- 件数上限の最大値（上限の上限）を超える値が指定された入力に対しては、利用者が状況を識別できる結果（入力エラー or 内部上限への切り詰め通知のいずれか。TBD-010 で確定）が返される
- §4.2 構造化結果のフィールド値（ファイルパス・行番号・マッチ行内容）は「マッチ詳細」テキスト出力と意味が一致する

**未確定事項への参照**: TBD-001（出力モード既定値の見直し）／TBD-006（件数上限既定値の見直し）／TBD-010（件数上限最大値超過時の挙動）

### 4.2 構造化された検索結果（search_code 等）

**要件**:
検索結果は、人間が読める形式（既存）に加えて、**フィールド名で識別可能な構造化データ**としても取得できる。

§4.2 は全対象ツール共通の最低限フィールドのみを定義する。各ツール固有の追加フィールドは §4.4 で個別に定義する（§4.5 は本 Feature ではスコープ外）。

**共通フィールド（全対象ツール最低限）**:

| フィールド | 内容 |
| ---------- | ---- |
| ファイルパス | マッチが見つかったファイルの絶対パス |
| 行番号 | マッチした行の番号（1-indexed） |
| マッチ行内容 | 行のテキスト内容（前後空白除去） |
| 検索結果総数 | 上限適用前の総数 |
| 省略フラグ | 件数上限による省略の有無 |

**ツール固有の追加フィールド**:

| 対象ツール | 追加フィールド | 定義箇所 |
| ---------- | -------------- | -------- |
| find_symbol_definition | 種別・所属スコープ情報（親スコープ名 / extension 対象型 / モジュール名） | §4.4 |

**対象ツール**:
- search_code
- find_symbol_definition（4.4 と連動）

**受入基準**:
- 既存のテキスト出力は維持される（後方互換）
- 構造化結果が同時に取得可能である
- 利用者がパースせずにフィールドへアクセスできる
- 共通フィールドは全対象ツールで取得できる
- ツール固有の追加フィールドは §4.4 で定義された内容を満たす

**注**: 提供方法（MCP の structuredContent 併記等）は設計フェーズで決定する（How）。

**未確定事項への参照**: TBD-004（構造化結果の表現形式）

### 4.3 ファイルパターンの複数指定（search_code）

**要件**:
`search_code` のファイル絞り込みは、**含めるパターン**と**除くパターン**をそれぞれ複数指定できる。

**典型的な利用シナリオ**:
- テスト・モック・自動生成コードを除外して本実装のみを検索
- 複数の拡張子・ディレクトリを同時対象にする

**指定方法**:
- 含めるパターン: 0 個以上の glob（空配列は未指定と等価）
- 除くパターン: 0 個以上の glob（空配列は未指定と等価）
- 既定: 含めるパターン未指定時は Swift ファイル全体（既存挙動）

**評価規則**:
- 含めるパターンに合致し、かつ除くパターンに合致しないファイルのみが対象となる
- 複数の含めるパターンは OR 結合
- 複数の除くパターンは OR 結合
- 含めるパターン・除くパターンの両方に合致するファイルは**除くパターンを優先**して除外する
- 不正な glob 構文を検出した場合は入力エラーとして利用者に明示する

**既存パラメータとの併存**:
- 既存の単一指定パラメータ（従来の `file_pattern` 相当）は維持し、後方互換として引き続き動作する
- 既存の単一指定と新規の含めるパターン配列が同時に指定された場合の優先順位は TBD-007 として §7 に記録し、設計フェーズで確定する
- 配列要素数の上限値（極端な指定でメモリ等を圧迫しないための上限）は要件レベルでは定めず、設計フェーズで上限方針を決める（要件側では下限 0 と既定挙動のみを規定する）

**受入基準**:
- 含めるパターン・除くパターンを配列で受け取れる
- 既存の単一指定（後方互換用）も引き続き動作する
- 既定の挙動は従来と一致する
- 不正な glob 入力時はエラーが利用者に明示される
- 含めるパターンと除くパターンが衝突した場合は除くパターンを優先する

**スコープ外**:
- 論理層による絞り込み機能は本 Feature に含めない（詳細は §5 を参照）

### 4.4 シンボル定義検索の絞り込み強化（find_symbol_definition）

**要件**:
`find_symbol_definition` は、同名シンボルが複数存在する場合に**種別**と**所属スコープ情報**で結果を絞り込み・識別できる。

#### 4.4.1 種別フィルタ

**指定可能な種別**（小文字スネーク表記。§3 用語定義に準拠）:
- struct / class / enum / protocol / actor
- function / variable / typealias / extension

**動作**:
- 種別を指定した場合、その種別のシンボルのみを返す
- 種別未指定時は全種別を返す（既存挙動を継承し、優先返却は REQ-003 §2.3 v0.6.5+ で定められた挙動を維持する）
- 未定義の種別文字列が指定された場合は入力エラーとして利用者に明示する

#### 4.4.2 所属スコープ情報の付与

**結果に含める情報（§3 用語定義「所属スコープ情報」の構成要素）**:

| 構成要素 | 内容 |
| -------- | ---- |
| 親スコープ名 | ネストされている親型の名前（存在する場合） |
| extension 対象型 | extension 内で定義された場合の対象型 |
| モジュール名 | 検出可能な場合 |

**区別すべきケース（受入基準で参照）**:
- (i) ルート定義の `Button`
- (ii) ネスト型 `enum Foo { struct Button }` の `Foo.Button`
- (iii) `extension Foo { struct Button }` の `Foo.Button`（extension 由来）

**受入基準**:
- 種別フィルタを指定できる
- 種別未指定時は REQ-003 §2.3 v0.6.5+ の優先返却を維持しつつ全種別を返す
- 未定義の種別が指定された場合はエラーが利用者に明示される
- 結果に**所属スコープ情報**（親スコープ名・extension 対象型・モジュール名）が含まれる
- 所属スコープ情報により、上記 (i)(ii)(iii) の 3 ケースを互いに区別可能である

**未確定事項への参照**: TBD-002（モジュール名取得の精度）／TBD-005（小文字スネーク指定値と既存表示用 kind 値の対応）

### 4.5 シンボル名による正確な参照検索 [本 Feature ではスコープ外]

**スコープ外の理由（歴史的経緯）**:

本節「シンボル名による正確な参照検索」は本 Feature ではスコープ外とする。理由は §6.3 に記録した歴史的経緯（`find_symbol_references` の意図的削除、`find_references` の未実装、`find_type_usages` の削除、Xcode プロジェクト環境での LSP 無効化等）による（詳細は §6.3 を参照）。

これらの実装判断が、本要件文書および REQ-002 / REQ-003 にフィードバックされないまま「シンボル名入力経路による LSP 精度参照検索」を本 Feature の要件として記述する状態になっていた。

**本 Feature での扱い**:

- 「シンボル名による正確な参照検索」機能は本 Feature では新設しない
- LSP 系参照検索ツール（`find_symbol_references` / `find_references`）の再導入可否は、それが本当に必要かという議論を含めて、別途独立 Feature として検討する
- 既存ツール（`find_symbol_definition`、`list_symbols` 等、SwiftSyntax 静的解析ベース）は本 Feature の他節（§4.1〜§4.4 / §4.6〜§4.8）の対象として維持する

**本節を将来再導入する場合に確定すべき事項**:

- LSP 系ツールが本当に必要か（過去の削除判断を覆す根拠）
- LSP 利用可否（Xcode 検出含む）に対するケーパビリティ通知設計（§4.7.1 の一般原則を再利用し、LSP 系ツール固有の前提条件・判定基準を §4.7.1 に追加する形で記述する）
- find_symbol_references / find_references の API 表面（既存ツール拡張 / 新規ツール追加 / 統合形）

**関連 TBD**: TBD-008（§4.5 既存前提ツールの本体実装スコープ）は本 Feature ではスコープ外確定により解消する（§7 から削除）。

### 4.6 後方互換性

**要件**:
既存の呼び出し方法・出力形式は、本 Feature 実装後も引き続き動作する。

**保証する後方互換**:
- 既存ツールの呼び出し時に新規パラメータを未指定でも、従来と同等の挙動を返す
- 既存の人間可読テキスト出力は維持される
- 既存ツール名・既存パラメータ名は変更しない

**「追加情報の混在」の境界（検証可能な範囲を構造的に限定）**:
- 既存出力の各行頭フォーマット（例: `<file>:<line>: <content>`）は変更しない
- 追加情報は**新規行・新規ブロック・末尾セクション**として付加する形でのみ許容する
- 既存行内に新たな区切り文字や追加トークンを混在させない

**受入基準**:
- 既存の呼び出しコードを変更せずに動作する
- 既存出力の行頭フォーマット（例: `<file>:<line>: <content>`）が変更されないことを確認できる
- 追加情報が新規行・新規ブロック・末尾セクションとしてのみ付加される
- Swift-Selena 自身の既存テストが破壊的変更なく通過する（既存テキスト出力に依存する自動テストを含む）
- 既存出力契約（テキスト出力の行頭フォーマット・既存ツールの呼び出し互換性）が本 Feature 完了後も維持されている

### 4.7 非機能要件

#### 4.7.1 ケーパビリティ通知

**要件**:
- MCP サーバは ListTools / list_available_tools の応答時点で、現在の環境（Xcode プロジェクト判定・ビルド可否・sourcekit-lsp 有無等、各ツールが必要とする前提条件）で動作可能なツールのみを返す
- 「動作可能」とは、当該ツールが必要とする前提条件をすべて充足しており、ツール呼び出し時に「環境前提未充足」を原因とするエラーを返さない状態を指す（前提条件の具体列挙および判定基準の詳細は設計フェーズで確定する）
- 動作不可能なツールはツール一覧に登録されない（クライアントから「存在しないツール」として見える）
- 利用者は initialize_project の応答および list_available_tools の応答から、現在の環境で利用可能なツール一覧（ケーパビリティ）を確認できる
- 動作可能でないツールに対する「暗黙の代替ツール降格」「内部フォールバック」は行わない（バグ隠蔽の温床になるため）
- 上記「暗黙の代替ツール降格」「内部フォールバック」非実施は、ListTools 応答に含まれないツール名がクライアントから呼び出された場合に「ツール未登録」エラーが返り、別ツールの結果が代理返却されないことで観察可能である（注 3）

**注 1**: 本 Feature では §4.5 でスコープ外として扱う LSP 系参照検索ツール（`find_symbol_references` / `find_references`）は本節の規範対象に含めない。本節は LSP 系ツール特化の規範ではなく、本 Feature 対象ツール（§4.1〜§4.4 / §4.6〜§4.8）および将来追加されるツールに対する一般原則として記述する。

**注 2**: ツール動作可能性の検出タイミング・検出結果のキャッシュ戦略は設計フェーズで決定する（要件レベルでは規定しない）。

**注 3**: 受入基準 5 項目目（暗黙の代替ツール降格・内部フォールバック非実施）は規範であり、6 項目目はその規範が満たされていることを観察するための検証手段である。両者を分離することで規範と観察事象の Yes/No 判定主体を明確化する。

#### 4.7.2 運用性

**要件**:
- 利用者が結果から状況（件数上限による省略の有無等）を把握できる
- 利用者は initialize_project / list_available_tools の応答から、現在の環境で利用可能なツール一覧（ケーパビリティ）を把握できる（§4.7.1 と整合）
- 不正入力（正規表現・glob・件数上限・種別フィルタ）に対して原因と修正案を含むエラーメッセージを返す（REQ-003 §2.3 v0.6.5+ の方針と整合）
- エラーメッセージは最低限「原因」と「修正案」の 2 要素を機械的に区別可能な形で含む（具体的なフィールド名は設計フェーズで決定する）

### 4.8 異常系要件

**要件**:
正常系と並行して、以下の異常系・境界条件における挙動を本 Feature の要件として確定する。

| 区分 | 状況 | 要件 |
| ---- | ---- | ---- |
| 入力検証 | 正規表現の構文エラー（§4.1 search_code の pattern） | 入力エラーとして原因と修正案を利用者に明示する |
| 入力検証 | glob の構文エラー（§4.3 含めるパターン・除くパターン） | 入力エラーとして原因と修正案を利用者に明示する |
| 入力検証 | 件数上限が 0・負数・整数以外（§4.1） | 入力エラーとして利用者に明示する（最小値は 1） |
| 入力検証 | 未定義のシンボル種別文字列（§4.4.1） | 入力エラーとして利用者に明示する |
| 入力検証 | 含めるパターンと除くパターンが両方未指定 | 既定挙動（含めるパターン未指定 = Swift ファイル全体）を適用する |
| 入力検証 | 含めるパターンと除くパターンが同一ファイルに同時マッチ | 除くパターンを優先する（§4.3 評価規則と整合） |
| 永続化 | キャッシュ破損・読み込み失敗 | キャッシュを無視して通常解析を実行し、利用者に状況を明示する（既存ツール挙動と整合） |
| 永続化 | 構造化結果のシリアライズ／デシリアライズ失敗 | テキスト出力のみ返却し、構造化結果欠落をフラグで利用者に通知する |
| シンボル解決 | 所属スコープ情報の取得失敗（モジュール名解決失敗等） | 所属スコープ情報を空で返し、検出失敗である旨を結果に明示する |
| ケーパビリティ判定 | 環境検出 / ケーパビリティ判定処理の失敗（例: sourcekit-lsp 検出が異常応答を返す等） | 当該前提条件に依存するツールを ListTools 応答から除外し、利用者が initialize_project / list_available_tools の応答から判定失敗の状況を識別できるよう明示する（具体的なフィールド・メッセージ形式は設計フェーズで確定する） |

**受入基準**:
- 上記表の各状況において、それぞれの「要件」列に記載した挙動が成立する
- 入力検証エラーは利用者がエラー内容を見て修正できるメッセージを含む

## 5. スコープ外（本 Feature では扱わない）

| 項目 | 理由 / 振り分け先 |
| ---- | ----------------- |
| シンボル名による正確な参照検索（旧 §4.5） | 歴史的経緯（§6.3 参照）により本 Feature では新設しない。LSP 系参照検索ツールの再導入可否は別途独立 Feature として検討する（詳細は §4.5 / §6.3 参照） |
| コメント・文字列リテラル除外検索 | LSP ではなく SwiftSyntax の責務。性能トレードオフが大きく、別 Feature として独立評価が必要 |
| 論理層による絞り込み（layer enum 等） | Selena がプロジェクトの層構造を理解する基盤を持たないため、別 Feature とする |
| ファイル本文読み取りツール | 本 Feature の対象外。標準 Read 系の責務として明確に分離する |
| 既存ツールのデフォルト破壊的変更 | 後方互換維持の方針に反する |
| 構造品質の数値目標化 | spec_priorities_spec §3 に従い禁止 |
