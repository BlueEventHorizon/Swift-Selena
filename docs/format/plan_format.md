# 計画書フォーマットルール

## 目的

要件定義から設計、実装までの全工程で要件を追跡可能にし、以下を実現します：

- **要件と実装の紐付け**: すべての要件がどのファイルで実装されているか明確化
- **影響範囲の明確化**: 要件変更時の影響箇所を即座に特定
- **タスクベース開発**: 各タスクが要件と紐付き、Agentによる自動実装が可能

## ID体系

### 要件ID
要件IDの体系と命名規則は[要件定義書フォーマットルール](docs/format/spec_format.md#要件ID体系)に定義されています。

**主な要件IDプレフィックス**：
- **APP-**: アプリケーション全体概要 (Application Overview) ※最上位要件
- **SCR-**: 画面要件 (Screens)
- **CMP-**: UIコンポーネント (Components)  
- **BL-**: ビジネスロジック (Business Logic)
- **NF-**: 非機能要件 (Non-Functional)
- **FNC-**: 機能要件 (Functions)
- **DM-**: データモデル (Data Models)
- **EXT-**: 外部インターフェース (External Interfaces)

### 設計ID
- **DES-**: 設計書 (Design Documents)
- 連番形式：DES-001, DES-002, DES-003...

## ワークフロー間でのID活用

```
1. 要件定義フェーズ (import_app_workflow.md / import_figma_workflow.md)
   ↓ 要件IDを付与（例: SCR-001）
   
2. 設計フェーズ (design_workflow.md)
   ↓ 要件IDを参照して設計
   ↓ 設計IDを付与（例: DES-001）
   
3. 計画フェーズ (planning_workflow.md)
   ↓ 要件トレーサビリティマトリクス作成（要件ID → 設計ID）
   ↓ 設計トレーサビリティマトリクス作成（設計ID → タスクID）
   ↓ タスク一覧作成
   
4. 実装フェーズ (task_execution_workflow.md)
   ↓ タスク単位でAgent実行（設計書を参照）
   
5. 完了確認
   タスクのチェックマークで進捗確認
```

## 計画書の構成 [MANDATORY]

計画書（`project/{feature}/plan/{feature}_plan.md`）は以下のセクションで構成されます：

1. **要件トレーサビリティマトリクス** - 要件IDから設計IDへの対応
2. **設計トレーサビリティマトリクス** - 設計IDからタスクIDへの対応
3. **タスク一覧** - 実装タスクの詳細
4. **改定履歴** - 計画書の変更履歴

**上記以外のセクションは追加しないこと。** 計画書の目的は「優先度の高いタスクを検出し実行する」ことであり、タスク実行に不要な情報（依存関係図、概要説明等）は記載しない。タスク実行に必要な情報はすべてタスク表の各列（設計ID、依存関係、必読等）に含める。

## 要件トレーサビリティマトリクス

要件IDから設計IDへの対応を管理します。

### マトリクス構造

```markdown
| ✓ | 要件ID | タイトル | 内容 | 設計ID |
|---|--------|---------|------|--------|
| ✅ | APP-001 | アプリ全体概要 | 連絡先管理アプリの基本仕様 | - |
| ✅ | SCR-001 | 連絡先リスト画面 | メイン画面、グループサイドバー付き | DES-001 |
| ☐ | FNC-001 | グループ管理 | グループの作成・編集・削除機能 | 設計書作成待ち |
| ✅ | BL-001 | データ同期・永続化 | 連絡先データの同期と保存 | DES-001 |
```

### 各列の説明

- **✓**: 設計書作成状態（✅完了、☐未完了）
- **要件ID**: 要件定義書で付与された一意のID
- **タイトル**: 要件の日本語タイトル
- **内容**: 要件の簡潔な説明
- **設計ID**: 対応する設計書のID（複数可、APP-001のような最上位要件は設計書なしで「-」、未作成時は「設計書作成待ち」）

### 各列の説明

- **設計ID**: 設計書の一意のID
- **タイトル**: 設計書の日本語タイトル
- **内容**: 設計の簡潔な説明
- **要件ID**: 関連する要件ID（複数可）
- **タスクID**: 生成されるタスクID（複数可、範囲表記可）

### TDD対応タスク順序 [MANDATORY]

Domain層Serviceの実装は、以下の順序でタスクを実行する：

#### 新規Service実装の場合
```
1. Entity/Enum/ValueObject定義（優先度57）
2. DataStore/Repository Protocol定義（優先度57）
3. DataStoreMock/RepositoryMock実装（優先度56）
4. Unit Test作成（優先度56）← テストを先に作成
5. Service実装（優先度55）← テストを通す実装
6. リファクタリング（優先度53）← テスト通過後の整理
```

#### 既存Service修正の場合
```
1. Unit Test追加（現在の振る舞いを確認するテストを作成）
2. Service修正
3. テスト確認
```

**重要**: Unit TestはService実装より**先に**作成する。これにより、期待する振る舞いを明確に定義してから実装に取り掛かることができる。

### タスクの粒度

タスクは以下の基準で分割します：

1. **1つのAgent実行で完結する単位**
   - 1つのService、1つのView/ViewModel、1つのDataStore実装など
   - 対応するMock実装（ServiceMock、DataStoreMock等）は別タスクとして分割
   - 複数のViewやServiceを1タスクにまとめない

2. **独立して実装・テスト可能な単位**
   - 他のタスクの完了を待たずに着手できる
   - この実装、または修正した結果、ビルド、テストが成功するすることが必須。その規模に抑える [MANDATORY]

3. **目安となる規模**
   - 1つのファイル、または密接に関連する2〜３つのファイル（View/ViewModel等）
   - やるべき内容は5〜10項目程度に抑える

4. **粒度の例**
   - ✅ 良い例：XXXXXService実装、XXXXXServiceMock実装、XXXXXListView実装、XXXXXDataStoreImpl実装、XXXXXDataStoreMock実装
   - ❌ 悪い例：全Service実装、全画面実装、Domain層全体実装、ServiceとそのMockを同一タスクに含める

### 用語解説

以下のデザインパターンおよび専門用語の説明：

- **Adapter（アダプター）**: 既存のクラスやフレームワーク（例：Contacts Framework）を、アプリ内部のインターフェースに適合させる設計パターン。外部システムとの接続部分。
- **Agent（エージェント）**: UIの振る舞いを自動制御するコンポーネント。例：PhoneDialAgentは電話番号入力時の発信判定を自動化。
- **Factory（ファクトリー）**: オブジェクトの生成を管理する設計パターン。依存性注入（DI）の基盤となる。
- **Mock（モック）**: テスト用の模擬実装。実際の処理の代わりにダミーデータを返す。
- **Repository（リポジトリ）**: データアクセスを抽象化する層。Domain層がInfrastructure層の詳細を知らずにデータ操作できる。
- **DataStore（データストア）**: データの永続化を担当する層。UserDefaults、CoreData等を使用。
- **Migration（マイグレーション）**: データベースやストレージの構造変更処理。アプリ更新時のデータ移行。
- **DesignToken（デザイントークン）**: デザインシステムの最小単位。色、フォント、間隔等の定数。
- **ViewModifier**: SwiftUIでViewの見た目や動作を変更する修飾子。
- **Converter（コンバーター）**: データ形式を変換する処理。EntityとDTO（Data Transfer Object）の相互変換等。

## 利点

### AIエージェントにとっての利点
- タスク単位で独立したAgent実行が可能
- 要件IDによる作業対象の明確化
- やるべき内容が明確で実装漏れを防止

### 人間にとっての利点
- タスクのチェックマークで進捗が一目瞭然
- 層ごとの表形式で全体像を把握しやすい
- 要件IDとタスクの紐付けで影響範囲が明確

## 要件IDの命名例

- **APP-001**: アプリケーション全体概要 (最上位要件定義書)
- **SCR-001〜**: 画面要件 (screens/)
- **CMP-001〜**: UIコンポーネント (ui_components/)
- **BL-001〜**: ビジネスロジック (business_logic/)
- **NF-001〜**: 非機能要件 (non_functional/)
- **FNC-001〜**: 機能要件 (functions/)

## 注意事項

1. **要件IDの不変性**: 一度付与した要件IDは変更しない
2. **削除時の扱い**: 要件が削除されても、IDは欠番として残す
3. **追加時の採番**: 新規要件は必ず最後の番号の次を使用
4. **カテゴリの厳守**: ディレクトリ構造とIDプレフィックスを必ず一致させる
5. **APP-001の特殊性**: アプリケーション全体概要は常にAPP-001として固定、設計書は不要
