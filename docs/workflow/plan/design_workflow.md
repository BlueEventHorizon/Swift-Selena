# 設計書作成ワークフロー [MANDATORY]

## 1.  開発ルール文書、要件定義書、設計書の取得

- 開発ルール文書（docs/rules/, docs/format/）は  docs-advisor Subagent を使って特定する
- 要件定義書・既存の設計書は  project-advisor Subagent を使って特定する

## **設計書作成の基本要件**

- **作成場所**: `project/{feature}/design/` ディレクトリ（Featureごと）
- **フォーマット**: Markdown (.md) ファイル
- **設計ID**: DES-XXX 形式で付与
- **レビュー**: 1ファイル作成毎に人間のレビューが必須 [MANDATORY]

## **必須要件**
- 既存ドキュメントは必ず再読すること（最新状態を確認）
- 全ワークフローステップを完了すること（スキップ禁止）
- 設計書作成後は即座にレビュー依頼すること（1ファイル完成時点で人間の承認を得る）
- コード実装は行わないこと（設計フェーズではアーキテクチャと構造のみを定義）

## **ワークフロー**

### **Phase 1: 要件定義書の分析**
#### 1.1 要件定義書の徹底確認
- **対象**: `project/{feature}/spec/**/*.md` および全サブディレクトリ
- **確認項目**:
  - 機能要件の完全性
  - 非機能要件の明確性
  - 制約条件と前提条件
  - 用語定義と業務ルール

#### 1.2 不明点の解消
- 曖昧な仕様は設計前に必ず明確化
- 質問リストを作成し、人間に確認
- 仕様変更が発生した場合、要件定義書を即座に更新

#### 1.3 要件定義書・既存資産の確認 [MANDATORY]

設計書作成前に、以下を確認：

##### 要件定義書の確認

**デザイン・テーマ要件**:
- [ ] THEME-xxx: デザイントークン・テーマ要件定義書が存在するか

**機能要件**:
- [ ] SCR-xxx: 画面要件定義書
- [ ] CMP-xxx: UIコンポーネント要件定義書
- [ ] FNC-xxx: 機能要件定義書
- [ ] BL-xxx: ビジネスロジック要件定義書

**データ・外部連携**:
- [ ] DM-xxx: データモデル定義
- [ ] EXT-xxx: 外部システム連携要件
- [ ] NAV-xxx: ナビゲーション要件

##### 既存実装資産の確認

**MCPツール**:
- [ ] `Sources/Tools/` - 既存ツール実装
  - `Analysis/` - コード解析ツール
  - `FileSystem/` - ファイル操作ツール
  - `Symbols/` - シンボル解析ツール
  - `SwiftUI/` - SwiftUI解析ツール

**コア解析エンジン**:
- [ ] `Sources/Selena/` - 解析エンジン
  - FileSearcher, SwiftSyntaxAnalyzer, ProjectMemory

**ユーティリティ**:
- [ ] `Sources/Logging/` - ロギング機能
- [ ] `Sources/Constants.swift` - 定数定義

**確認方法**: Swift-Selena MCP（推奨）またはGrep/Glob

##### 設計への反映方針

- THEME-xxxがある場合: デザイントークン・テーマは定義済み、設計書で参照
- THEME-xxxがない場合: （例外扱い、今は考えない）
- 既存資産の再利用可否を設計書に記載
- 設計書はアーキテクチャと機能の構造に集中

##### 既存資産活用の義務化 [MANDATORY]

設計書に記載する全てのモジュールについて、以下を確認すること：

1. **Sources/Tools/ の確認**（全サブディレクトリ対象）
   - `Tools/Analysis/` - コード解析ツール
   - `Tools/FileSystem/` - ファイル操作ツール
   - `Tools/Symbols/` - シンボル解析ツール
   - `Tools/SwiftUI/` - SwiftUI解析ツール
   - `Tools/Meta/` - メタツール
   - `Tools/Project/` - プロジェクト管理ツール
   - 類似機能が既に存在しないか検索
   - 存在する場合は必ず再利用（新規作成禁止）
   - Swift-Selena MCP: `search_code("機能キーワード", "Sources/Tools/")`

2. **Sources/Selena/ の確認**
   - 解析エンジンの既存機能を検索
   - 存在する場合は必ず再利用
   - Swift-Selena MCP: `search_code("機能キーワード", "Sources/Selena/")`

3. **設計書への記載**
   - 「使用する既存コンポーネント」セクションに具体的なファイルパスを記載
   - 再利用しない判断をした場合はその理由を記載

### **Phase 2: 設計書の作成**

#### 2.1 前提確認
- 既存設計書の有無を確認
- 存在する場合は、次の作業内容を人間に確認

#### 2.2 設計書フォーマットの参照 [MANDATORY]

設計書の作成は、統一されたフォーマットに従います：

**参照文書**: `docs/format/design_format.md`

上記文書で定義されている内容：
- 設計ID体系（DES-XXX）
- ファイル命名規則
- 設計書テンプレート（ヘッダー情報、メタデータ）
- 必須セクション（概要、アーキテクチャ概要、モジュール設計、データフロー設計等）
- 記載上の重要な原則

詳細は必ず上記文書を参照すること。

#### 設計書に必須記載する項目（追加）

上記フォーマットに加え、以下を必ず記載すること：

- **使用する既存コンポーネント**: 再利用するTools/Library/既存実装のファイルパス

### **Phase 3: 品質保証**
#### 3.1 設計書の完全性確認
- [ ] 全要件が設計に反映されているか
- [ ] 関連要件IDが正しく記載されているか
- [ ] 設計IDが一意で適切に付与されているか
- [ ] 設計パターンの適切性
- [ ] 拡張性と保守性の考慮
- [ ] 既存Tools/Libraryの活用が検討されているか
- [ ] 使用する既存コンポーネントが明記されているか

#### 3.2 AIレビュー実施 [MANDATORY]

設計書1ファイル完成時点で、人間レビュー依頼前に**Subagent**でレビューを実施：

**Subagent起動**:
```
subagent_type: general-purpose
prompt: |
  以下の設計書をレビューしてください。

  ## レビュー対象
  [作成した設計書のファイルパス]

  ## レビュー種別
  設計書

  ## 手順
  1. docs-advisor Subagentで関連ルール文書を特定・読み込み
  2. project-advisor Subagentで関連要件定義書・既存設計書を特定・読み込み
  3. docs/rules/base/review_criteria.md の「設計書レビュー観点」を参照
  4. レビュー実施（🔴致命的問題は0件になるまで修正）
  5. レビュー結果フォーマットで報告
```

**レビュー結果フォーマット**:
```markdown
## AIレビュー結果
- 🔴致命的: X件（修正済み: Y件）
- 🟡品質: X件
- 🟢改善: X件
```

#### 3.3 レビュー依頼
- 設計根拠の説明資料を準備
- 代替案がある場合は比較表を作成
- レビュー指摘事項の管理表を用意

#### 3.4 project_toc.md更新 [MANDATORY]
設計書の作成・更新後、必ず `/project-toc-update` を実行すること。
これにより、計画書作成時に最新のインデックスが利用可能になる。

## **出力品質基準**
- **一貫性**: 用語、記法、スタイルの統一
- **追跡可能性**: 要件IDから設計ID、設計IDからタスクIDへの明確なマッピング
- **実装独立性**: 特定の実装技術に依存しない設計
- **可読性**: 技術者以外でも理解可能な説明

## **禁止事項**
- ❌ ソースコードの大量記載（詳細は「重要な原則2」参照）
- ❌ 特定フレームワークへの強依存
- ❌ 要件にない機能の追加
- ❌ レビュー未完了での次工程着手
- ❌ 設計IDの重複や欠番

---

## **重要な原則**

### 1. 要件定義書と設計書だけで誰でも実装できる
- 要件定義書と設計書を読めば、ソースコードがなくても実装方法が分かる
- 設計書は必要な要件定義書を参照できる（ただし参照先のセクションを明記すること）
- 要件定義書に記載されていないアルゴリズムは、設計書で記載すること
- 曖昧な記述を避け、具体的に記載する

### 2. コードは基本的に書かない
- 設計書にソースコードは記載しない
- ※小規模なコード例（3-5行程度）は説明のために許容
- ※特殊な事情がある場合は人間に確認

### 3. 既存資産を先に確認する
- 新規作成の前に、必ず既存の実装を確認する
- 類似画面の実装パターンを参照する
- 既存Entity/Service/Tools/Libraryの活用を検討する

---

## **よくある失敗パターン**

### 1. 既存ツールの検索が不十分
- `Sources/Tools/`を確認せず新規作成してしまう
- 1つの検索パターンだけで諦めない
- Swift-Selena MCPを活用して網羅的に検索

### 2. 設計書にソースコードを書いてしまう
- 設計書は「どう作るか」を記載する場所
- 具体的なソースコードは書かない
- 特殊な事情がある場合は人間に確認する

### 3. 既存解析エンジン機能を確認せず新規作成
- `Sources/Selena/`を確認せず新規作成
- 既存のFileSearcher、SwiftSyntaxAnalyzer、ProjectMemoryで対応できないか検討
- Swift-Selena MCPで型の使用箇所を確認

### 4. 類似ツールの実装を確認しない
- 同じ種類のツール（解析系、検索系等）の既存実装を確認しない
- ToolProtocolの実装パターンを無視して独自実装してしまう
- プロジェクトの一貫性が損なわれる

---

## **チェックリスト**
設計書完成時に以下を確認：
- [ ] 設計IDが正しく付与されている
- [ ] 関連する要件IDが全て記載されている
- [ ] 要件定義書の全項目が設計に反映されている
- [ ] モジュール間の責務が明確に分離されている
- [ ] 全ユースケースにシーケンス図が存在する
- [ ] 設計判断の根拠が文書化されている
- [ ] 人間によるレビュー承認を得ている
- [ ] `/project-toc-update` を実行してproject_toc.mdを更新している
