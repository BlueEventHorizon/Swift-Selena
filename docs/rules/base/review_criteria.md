# レビュー観点定義

各フェーズのAIレビューで使用する観点を定義します。

## 共通ルール

### カテゴリ定義
| カテゴリ | 意味 | 対応 |
|---------|------|------|
| 🔴致命的 | 重大な問題、即座に修正必須 | 0件になるまで修正を繰り返す |
| 🟡品質 | 品質上の問題、修正推奨 | 可能な限り修正 |
| 🟢改善 | 改善提案、対応任意 | 報告のみ |

### レビュー結果フォーマット
```markdown
## AIレビュー結果

### 🔴致命的問題
1. **[問題名]**: [具体的な説明]
   - 箇所: [ファイル名/セクション名]
   - 修正: [修正内容]

### 🟡品質問題
1. **[問題名]**: [具体的な説明]

### 🟢改善提案
1. **[提案名]**: [具体的な説明]

### サマリー
- 🔴致命的: X件（修正済み: Y件）
- 🟡品質: X件
- 🟢改善: X件
```

---

## Subagent実行方式 [MANDATORY]

AIレビューは**Subagent**で実行すること。これにより：
- クリーンなコンテキストで客観的なレビューが可能
- 文書作成時のバイアスを排除
- 関連文書を体系的に読み込んでレビュー

### レビューSubagentの起動方法

```
subagent_type: general-purpose
prompt: |
  以下の文書をレビューしてください。

  ## レビュー対象
  [対象ファイルパス]

  ## レビュー種別
  [要件定義書 / 設計書 / 計画書 / コード]

  ## 手順
  1. docs-advisor Subagentで関連ルール文書を特定・読み込み
  2. project-advisor Subagentで関連要件定義書・設計書を特定・読み込み
  3. docs/rules/base/review_criteria.md の該当セクションを参照
  4. レビュー実施
  5. レビュー結果フォーマットで報告
```

### レビュー種別ごとの関連文書

| レビュー種別 | docs-advisorで取得 | project-advisorで取得 |
|-------------|-------------------|---------------------|
| 要件定義書 | spec_format.md, architecture_rule.md | 関連する他の要件定義書 |
| 設計書 | design_format.md, architecture_rule.md, 層別ルール | 関連要件定義書、既存設計書 |
| 計画書 | plan_format.md | 関連要件定義書、設計書 |
| コード | architecture_rule.md, 層別ルール | 関連要件定義書、設計書 |

### 重要な原則

1. **必ずadvisorを使う**: 関連文書の特定はadvisor Subagentに任せる
2. **全文書を読む**: 特定された文書は全て読み込む（部分読みしない）
3. **観点に沿う**: このドキュメントのレビュー観点に厳密に従う
4. **客観的に評価**: 作成者の意図を推測せず、文書・コードの内容のみで判断

## 1. 要件定義書レビュー観点

要件定義書（`project/{feature}/spec/**/*.md`）のレビュー時に使用。

### 🔴致命的
| 観点 | 説明 | 例 |
|------|------|-----|
| What/Howの混同 | 実装方法（How）が混入している | 「SwiftUIのListを使用する」「Serviceクラスを作成する」 |
| 曖昧表現 | 検証不可能な表現 | 「適切に」「必要に応じて」「など」「等」 |
| 要件漏れ | 必須要件が欠落 | エラー処理、空状態、ローディング状態の未定義 |
| 矛盾 | 要件間の矛盾 | 同じ項目に対する異なる仕様 |

### 🟡品質
| 観点 | 説明 | 例 |
|------|------|-----|
| 要件ID不整合 | IDの重複・欠番・誤参照 | SCR-001が2つ存在、存在しないBL-999への参照 |
| 相互参照エラー | リンク切れ、参照先不一致 | 存在しないファイルへのリンク |
| フォーマット違反 | `docs/format/spec_format.md` 違反 | 必須セクションの欠落、命名規則違反 |
| 用語不統一 | 同じ概念に異なる用語 | 「ユーザー」と「利用者」の混在 |

### 🟢改善
| 観点 | 説明 | 例 |
|------|------|-----|
| 表現の明確化 | より明確な表現への改善 | 具体的な数値、条件の明示 |
| 図表追加 | 理解を助ける図表の提案 | 画面遷移図、状態遷移図 |
| 構成改善 | 読みやすさの向上 | セクション分割、見出し追加 |

## 2. 設計書レビュー観点

設計書（`project/{feature}/design/**/*.md`）のレビュー時に使用。

### 🔴致命的
| 観点 | 説明 | 例 |
|------|------|-----|
| アーキテクチャルール違反 | `docs/rules/base/architecture_rule.md` 違反 | 層間の不正な依存、責務の逸脱 |
| 要件未反映 | 要件定義書の項目が設計に未反映 | 要件で定義された機能の設計欠落 |
| Entity必須Protocol欠落 | Sendable/Equatable/Identifiable/Codable/Hashable | 必須プロトコルの準拠漏れ |
| Service設計違反 | `public actor` でない、データ保持 | classでの実装、キャッシュの保持 |

### 🟡品質
| 観点 | 説明 | 例 |
|------|------|-----|
| DataStore設計不備 | Protocol設計違反 | getCurrentItems/itemsStream/updateItems欠落 |
| データフロー不整合 | Service→DataStore→ViewModelの流れ違反 | ViewModelからRepository直接参照 |
| DI設計欠落 | Factory登録の設計漏れ | 新規ServiceのServiceFactory登録忘れ |
| ViewModel設計不備 | @MainActor @Observable欠落、ViewState未使用 | 個別Boolフラグの設計 |

### 🟢改善
| 観点 | 説明 | 例 |
|------|------|-----|
| 既存資産活用 | Tools/Library/既存実装の再利用提案 | 類似コンポーネントの指摘 |
| パターン統一 | プロジェクト内パターンとの整合性 | 命名規則、構造の統一 |
| 拡張性考慮 | 将来の拡張を見据えた設計提案 | インターフェース分離 |

## 3. 計画書レビュー観点

計画書（`project/{feature}/plan/**/*.md`）のレビュー時に使用。

### 🔴致命的
| 観点 | 説明 | 例 |
|------|------|-----|
| 要件反映漏れ | 要件トレーサビリティマトリクスの欠落 | 要件定義書に存在する要件が未記載 |
| 設計反映漏れ | 設計トレーサビリティマトリクスの欠落 | 設計書に存在する設計が未記載 |
| 依存関係循環 | タスク間の循環依存 | A→B→C→Aのような循環 |
| タスクID重複 | 同じIDのタスクが複数存在 | D-001が2つ存在 |

### 🟡品質
| 観点 | 説明 | 例 |
|------|------|-----|
| タスク粒度不適切 | 1Agent実行で完結しない規模 | 複数機能を1タスクに詰め込み |
| 優先度設定不備 | 優先度ルール違反 | Domain層タスクが30（UI層の値） |
| 依存関係不明確 | 先行タスクの指定漏れ | 明らかな依存があるのに未指定 |
| やるべき内容不足 | タスク内容が曖昧 | 「実装する」のみで詳細なし |

### 🟢改善
| 観点 | 説明 | 例 |
|------|------|-----|
| 並列実行可能タスク | 依存関係がないタスクの特定 | 独立したEntity定義タスク群 |
| タスク分割提案 | より細かい分割の提案 | 大きなタスクの分割案 |
| 効率化提案 | 実装順序の最適化 | 共通コンポーネント優先 |

## 4. コードレビュー観点

ソースコード（`App/`, `Domain/`, `Infrastructure/`, `DI/`, `Tools/`）のレビュー時に使用。

### 🔴致命的
| 観点 | 説明 | 例 |
|------|------|-----|
| コンパイルエラー | ビルド失敗を引き起こすコード | 構文エラー、型不一致 |
| 実行時例外 | クラッシュを引き起こすコード | 強制アンラップ、範囲外アクセス |
| セキュリティ脆弱性 | セキュリティ上の問題 | ハードコードされた認証情報 |
| アーキテクチャ違反 | 層間依存ルール違反 | UI層からRepository直接参照 |

### 🟡品質
| 観点 | 説明 | 例 |
|------|------|-----|
| 保守困難 | 理解・修正が困難なコード | 過度に複雑な条件分岐 |
| 拡張困難 | 変更・追加が困難な構造 | 密結合、ハードコード |
| テスト困難 | テスト作成が困難なコード | 依存性注入なし、副作用 |
| ルール違反 | `docs/rules/**/*.md` 違反 | コーディング規約違反 |

### 🟢改善
| 観点 | 説明 | 例 |
|------|------|-----|
| 最適化可能 | パフォーマンス改善の余地 | 不要な再計算、メモリ効率 |
| 慣用法違反 | Swift/SwiftUIの慣用法から逸脱 | より簡潔な記法の提案 |
| 文書化不足 | コメント・ドキュメント不足 | 複雑なロジックの説明欠落 |

## 問題数制限

レビュー時の問題数上限（過剰な指摘を防止）:

| カテゴリ | 上限 | 超過時の対応 |
|---------|------|-------------|
| 🔴致命的 | 10件 | 超過分は次回レビューへ |
| 🟡品質 | 10件 | 超過分は次回レビューへ |
| 🟢改善 | 5件 | 超過分は省略 |

---

## レビュー実施手順

1. 該当するレビュー観点セクションを参照
2. 対象文書/コードを観点に沿ってチェック
3. 発見した問題をカテゴリ別に分類
4. 🔴致命的問題を即座に修正
5. レビュー結果フォーマットでサマリーを報告
6. 🔴が0件になるまで修正→レビューを繰り返す
