## 1. 新旧バージョンの判定

ファイルの「新しさ」は以下の優先順位で判定する:

| 優先度 | 判定材料 | 例 |
|--------|----------|-----|
| 1 | ファイル名のバージョン番号 | `spec_v2.0.md` > `spec_v1.0.md` |
| 2 | ファイル内のバージョン記述 | `## バージョン: 2.0` |
| 3 | git log の最終コミット日時 | より新しいコミット |
| 4 | ファイル内の更新履歴 | より後の日付 |

**注意**: ファイル名の連番（001, 002...）は作成順であり、新旧を示すとは限らない。

## 2. 要件の分類判定

各要件を「既存要件の強化」か「完全新規追加」かに分類する。

### 既存要件の強化

以下のいずれかに該当する場合:
- ベースファイルに同じ機能名・セクション名が存在する
- タイトルが「〜の強化」「〜の改善」「〜の精度向上」等
- 既存セクションの詳細化・条件追加

**処理**: ベースファイルの既存セクションに内容を追記/更新する。

### 完全新規追加

以下のいずれかに該当する場合:
- ベースファイルに該当するセクションが存在しない
- 全く新しい機能・要件カテゴリ

**処理**: ベースファイルに新セクションとして追加する。

## 3. 重複検出パターン

### パターンA: 同一IDまたは同一機能

同じ機能について複数ファイルに記述がある場合:
→ 両方を読み比べ、より詳細・より新しい内容に更新。情報を失わないようにマージ。

### パターンB: 同一カテゴリの数値差異

```
ファイルA: パフォーマンス要件 — レスポンス3秒以内
ファイルB: パフォーマンス要件 — レスポンス2秒以内
```

→ より厳しい/より新しい基準を採用し、理由を記載。

### パターンC: 前文・概要の重複

複数ファイルにプロジェクト全体の概要が記述されている場合:
→ 最も詳細な記述をベースとし、他は削除。

---

## 4. 矛盾検出パターン

### パターンA: 数値の不一致

同じ指標に対して異なる数値が設定されている場合:
→ ユーザーに確認。通常は新しいファイル・より厳しい要件を採用。

### パターンB: 機能の必須/任意の不一致

```
ファイルA: 機能Xはオプション
ファイルB: 機能Xは必須
```

→ ユーザーに確認必須。仕様変更の経緯を確認する。

### パターンC: アーキテクチャ記述の不一致

ファイル間でアーキテクチャの前提が異なる場合:
→ 最新の実装状態（ソースコード）と照合してユーザーに確認。

---

## 5. 多ファイル統合の順序

3ファイル以上を統合する場合の推奨順序:

1. **ベースファイルを決定**: 最も包括的・最も多くの要件を含むファイル
2. **優先度順にマージ**: 新しいファイルから順番に処理
3. **段階的に確認**: 1ファイルずつマージして確認を得る

---

## 6. IDの整合性管理

### 統合前

対象ファイル群で使われているID体系（プレフィックス、番号規則）を把握する。

### 統合後の確認

- 同一IDが複数箇所に残っていないか確認
- 削除した要件のIDは欠番として残す（再利用しない）
- 新規追加要件のIDはベースファイルの体系に従って付番

---

## 7. 統合後チェックリスト

統合完了前に以下を確認する:

- [ ] 吸収元ファイルの全要件がベースファイルに含まれている
- [ ] IDの重複がない
- [ ] 矛盾する記述が残っていない（解消済み）
- [ ] 成功基準・受入条件テーブルが更新されている
- [ ] バージョン履歴セクションが存在する場合は更新されている
- [ ] 目次・インデックスの更新が必要かどうか確認した
